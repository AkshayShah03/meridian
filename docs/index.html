The Problem

AI assistants are fundamentally broken

Most AI chat products share the same five failure modes. Meridian is engineered specifically to eliminate each one.

🌀
Hallucination
LLMs confidently fabricate facts, numbers, and current events. There's no ground truth enforcement.
🕰️
Stale Knowledge
Training cutoffs mean models can't answer "What's happening now?" or compute today's date.
🧠
No Long-Term Memory
Every session starts blank. Users repeat themselves. Context is lost across conversations.
🔮
Zero Observability
When something breaks — which model, which token, which service? Black box with no trace.
🧱
Tight Coupling
Monolithic AI APIs mean swapping a model or adding a feature requires rewriting the whole backend.
The Solution

Meridian's five-layer answer

Each design decision maps directly to one of the problems above.

01 · ANTI-HALLUCINATION
Tool-First Routing
Factual queries never reach the LLM alone. The ReAct agent loop calls authoritative tools first — Google News RSS for headlines, IANA tzdata for time, an AST-based calculator for arithmetic, SSRF-protected URL fetch for web content. The LLM only synthesises natural language around verified data.
ReAct LoopTool RuntimeAST Calculator
02 · REAL-TIME DATA
Live Source Integration
The tool runtime fetches live data on every request. News is pulled from Google News RSS at query time. Timezone data is sourced from the IANA tzdata package, not a model weight. Date/time is always accurate.
Google News RSStzdataDuckDuckGo Search
03 · SEMANTIC MEMORY
Multi-Layer Memory Architecture
Three memory layers stack on top of each other: Redis short-term history for the current session, pgvector for semantic similarity recall across all past conversations, and episodic summaries to compress long histories without losing meaning. The memory service handles all of this over Kafka — fully decoupled from inference.
pgvectornomic-embed-textRedis
04 · FULL OBSERVABILITY
Five-Layer Tracing Stack
Every request generates traces through OpenTelemetry → Jaeger. Metrics flow to Prometheus → Grafana. LLM-specific traces go to Langfuse. And an independent eval-service automatically scores each response for coherence, relevance, completeness, latency, and safety — writing scores to Postgres.
OpenTelemetryJaegerLangfuseGrafana
05 · ZERO COUPLING
Kafka-Native Event Bus
No service calls another service directly. Every interaction — user events, inference requests, tool calls, eval scores, audit log — flows through named Kafka topics. Swap Ollama for GPT-4, add a new tool, or replace the eval algorithm: zero other services change.
Apache Kafka12 Topicsaiokafka
BONUS · EFFICIENCY
Prompt Caching + Multi-Model Routing
The inference gateway caches prompt prefixes in Redis. Identical or near-identical system prompts skip re-tokenisation, cutting TTFT significantly on repeated queries. Model routing selects llama3.2:3b for simple queries, mistral:7b for complex ones, and Anthropic Claude as a premium fallback.
Prompt Cachellama3.2:3bmistral:7bClaude

Request Flow

A message, step by step

What happens when you ask "What time is it in Tokyo right now?" with agent mode on.

1
Browser → WS Gateway
Message arrives over WebSocket. The JWT is validated inline — no auth service call, stateless verification.
ws://gateway:8002
2
Gateway → Kafka user-events
The gateway publishes the message to the user-events topic with the session ID. No service knows who's listening.
3
Memory Service recalls context
Memory service consumes the event, retrieves recent Redis history and runs a pgvector similarity search against past conversations to surface relevant episodic context.
nomic-embed-text · cosine similarity · pgvector
4
Agent Orchestrator runs ReAct loop
The orchestrator detects "time" in the query via keyword pre-router and dispatches a tool call to the tool-requests topic. It stores intermediate reasoning state in Redis.
ReAct: Reason → Act → Observe → Reason …
5
Tool Runtime executes datetime lookup
Tool runtime reads from IANA tzdata (bundled, always accurate) and returns the exact local time in Tokyo. No LLM involved in the lookup — zero hallucination risk.
tzdata · IANA database · Python zoneinfo
6
Inference Gateway synthesises & streams
The verified tool result is injected into the prompt. The gateway checks the Redis prompt cache first. On a miss, it streams token-by-token from Ollama. Tokens publish to llm-responses in real time.
Prompt cache hit → 0 inference · Miss → Ollama llama3.2:3b stream
7
WS Gateway fans out to browser
The gateway consumes the response stream from Kafka and pushes each token over WebSocket to the exact session that sent the original message.
8
Eval Service scores the response
After streaming ends, the eval service asynchronously scores coherence, relevance, completeness, latency, and safety. Results are written to Postgres for trend analysis and model comparison.
5 heuristic dimensions · Postgres audit log · Grafana dashboards
